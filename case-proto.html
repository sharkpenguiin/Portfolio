CASE 03 · 프로토타입 데모

텍스트 스펙 대신
동작하는 화면으로 전달

인터랙티브 재현 샘플동작하는 화면 = 스펙jsdom 자동화 검증

문서로 “이렇게 동작합니다”라고 쓰는 대신, 실제로 눌러볼 수 있는 화면으로 기획 의도를 전달합니다. 아래 데모는 마우스로 직접 조작해볼 수 있습니다. 가상 도메인으로 만든 재현 샘플이며 실제 산출물이 아닙니다.

WHY왜 문서가 아니라 화면인가

같은 기획을 전달하는 두 가지 방법. 오른쪽이 해석의 여지를 줄입니다.

문서 스펙
“필수값이 없으면 임시저장으로 처리한다”

읽는 사람마다 ‘임시저장’의 동작을 다르게 상상합니다. 버튼 위치, 배지 표시, 목록 반영 방식은 결국 개발 단계에서 다시 질문으로 돌아옵니다.

동작하는 화면
눌러보면 바로 안다

임시저장을 누르면 목록에 ‘임시’ 배지가 붙는다 — 설명이 필요 없습니다. 엣지 케이스(필수값 누락 시 등록 차단)도 직접 체험되므로 합의가 빨라집니다.

DEMO거래처 관리 · 직접 조작해보세요

항목을 클릭하면 상세가 열리고, ‘거래처 등록’으로 새 항목을 추가할 수 있습니다. 등록 시 사업자번호가 없으면 차단되고, 임시저장은 필수값 없이도 ‘임시’ 상태로 추가됩니다.

거래처 관리 — 가상 도메인 (유통 ERP)인터랙티브 재현 샘플
3개 거래처 · 클릭해서 상세 보기
    왼쪽 목록에서 거래처를 선택하세요.

    ※ 데이터는 브라우저 메모리에만 저장되며 새로고침하면 초기화됩니다.

    검증프로토타입도 테스트로 지킨다

    동작하는 화면이라면 동작을 자동으로 검증할 수 있습니다. 위 데모의 핵심 규칙을 jsdom 기반 테스트로 고정해, 화면을 수정해도 규칙이 깨지지 않게 했습니다.

    WHY · 왜 테스트까지프로토타입은 “한 번 보여주고 버리는 것”으로 취급되기 쉽지만, 기획 의도가 담긴 규칙(필수값 검증·임시저장 동작)은 이후 실제 개발의 기준이 됩니다. 테스트로 규칙을 코드화해 두면, 화면이 바뀌어도 “무엇이 맞는 동작인지”가 흔들리지 않습니다.
    // prototype.test.js — jsdom 자동화 검증 (발췌)
test('등록 시 사업자번호가 없으면 저장을 막는다', () => {
  openModal();
  fill('#f-name', '가상상사');      // 사업자번호는 비움
  click('#btnSave');
  expect($('#err').hidden).toBe(false);   // 에러 노출
  expect(rows().length).toBe(3);          // 목록 불변
});

test('임시저장은 필수값 없이도 항목을 추가한다', () => {
  openModal();
  fill('#f-name', '임시 거래처');
  click('#btnDraft');
  expect(rows().length).toBe(4);          // 1건 추가
  expect(lastBadge()).toBe('임시');       // 임시 배지
});
    STEP 1
    화면 제작

    기획 의도를 실제 동작하는 HTML/JS 화면으로 구현.

    STEP 2
    규칙 테스트화

    필수값 검증·임시저장 등 핵심 규칙을 jsdom 테스트로 고정.

    STEP 3
    회귀 방지

    화면을 수정해도 규칙이 깨지면 테스트가 실패로 알림.

    기획도 “돌려서” 검증할 수 있습니다 — 프로토타입을 테스트 가능한 형태로 만들면, 기획 의도가 말이 아니라 실행 가능한 기준으로 남습니다. 이 포트폴리오 사이트 자체도 같은 방식으로 자동 검증하며 만들었습니다.
