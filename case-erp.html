CASE 01 · 제약 유통 B2B ERP 구축 (외주)

회의마다 요구사항이 바뀌는 프로젝트에서,
기획을 “체계”로 만들어 통제한 기록

거래처 5개 유형18개 메뉴병행 개발 환경외부·내부 회의 20회+요구정의 → 명세 → 핸드오프 단독 수행
1
문제

거래처 유형마다 요구·정책이 다르고, 매주 회의에서 요구사항이 갱신됨. 문서가 회의 속도를 못 따라가면 개발이 서로 다른 버전을 보고 만들게 되는 구조.

2
판단

사람이 부지런해지는 대신 체계가 기억하게 한다 — 화면 단위 PRD(SCR-ID), 메뉴 단위 결정로그, 파라미터 표를 설계해 “최신 확정본”이 항상 한 곳에 있게 함.

3
결과

명세 작성 시 과거 회의록 전체를 재검토할 필요가 사라지고, Cross-functional 팀이 같은 기준을 참조. 해석 차이·재작업 리스크를 구조적으로 차단.

01 · PRD 체계화면 단위 SCR-ID 구조

요구사항 → 정책 → 필드가 한 흐름으로 이어지도록 화면(Screen) 단위로 문서를 짰습니다. 모든 요구사항에는 출처 태그가 붙습니다.

WHY · 왜 이렇게 설계했나메뉴 단위 문서는 커지면 아무도 안 읽게 되고, 기능 목록은 “왜”가 사라집니다. 개발자가 실제로 만드는 단위는 화면이므로, 화면 1개 = 문서 1섹션(SCR-ID)으로 맞추면 핸드오프에서 해석이 갈릴 틈이 줄어듭니다. 출처 태그는 “누가 원한 것인지”를 문서 시스템에 박아, 고객 요청과 PM 제안을 구분해 근거 없는 약속을 방지합니다.
SCR-CLI-001 · 거래처 관리 — 등록 화면 (가상 도메인: 유통 ERP)재현 샘플
구분요구사항출처정책 → 필드 연결
등록사업자번호 입력 시 중복·휴폐업 여부를 한 번에 확인현업 요청검증 버튼 1개로 3중 체크 → biz_no (필수·검증)
등록필수값이 없어도 임시 상태로 저장 가능업무 기반 해석상태값 draft 추가 → 목록에 “임시” 배지
조회거래 중단 거래처도 이력 조회는 가능해야 함현행 시스템삭제 = 비활성화 정책 → status (물리삭제 없음)
수정단가 변경 시 변경 사유 입력을 강제PM 제안변경 이력 테이블 분리 → price_log
공통담당자 이관 시 과거 실적 귀속 기준현업 확인미결 D-12 — 고객 확정 전 개발 보류
출처 태그로 현업 요청 원본 요구 · 업무 기반 해석 업무 맥락 기반 PM 해석 · 현행 시스템 기존 제약 · PM 제안 능동 제안 · 현업 확인 미결 → 확정 전까지 개발 보류

02 · 결정로그회의 20회를 이기는 문서

회의록은 시간순으로 두고, 결정만 메뉴(IA) 단위로 누적하는 별도 로그를 운영했습니다. 명세를 쓸 때 과거 회의록을 다시 뒤질 필요가 없어집니다.

WHY · 왜 이렇게 설계했나같은 주제가 여러 회의에 걸쳐 뒤집히면, “최신 결정이 무엇인가”를 찾는 비용이 기획 속도를 잡아먹습니다. 회의록(시간축)과 결정로그(메뉴축)를 분리하면 이력과 현재 상태를 둘 다 잃지 않습니다 — 나중에 “왜 이렇게 됐지”는 회의록에서, 지금 뭘 만들지는 결정로그에서.
메뉴별 결정로그 — 필터로 좁혀보세요재현 샘플
메뉴결정출처상태
거래처 관리삭제는 비활성화로 처리, 물리삭제 금지 (정산·이력이 참조)〔0512 외부〕확정
거래처 관리임시 저장 도입 — 필수값 없이 저장, 목록에 배지 표시〔0519 외부〕확정
정산 관리정산 단위: 거래처×품목×월 최소 단위로 적재 후 합산〔0526 외부〕확정
정산 관리수수료 상한 적용 방식 — 구간별 차등 여부〔0526 외부〕미결 D-07
품목 관리코드 체계: 관리코드와 표준코드 1:N — 데이터 대조로 확정〔0602 검증〕확정
품목 관리단종 품목 노출: 검색 시 기본 제외, 토글로 포함〔0609 내부〕현업 확인
요구사항은 그 메뉴의 회의에서만 나오지 않습니다 — 예컨대 거래처 화면의 핵심 요구가 정산 논의 중에 튀어나오는 식입니다. 그래서 메뉴별 문서를 쓸 때는 해당 회의만 보지 않고, 전체 회의 기록에서 그 주제를 교차 스윕해 다른 맥락에서 나온 언급까지 수집한 뒤 작성했습니다. 결정로그가 메뉴축인 이유도 이 흩어진 결정들을 한 곳에 모으기 위해서입니다.

03 · 파라미터 표“빈칸이면 개발이 알아서 채운다”

동작을 좌우하는 기준값·임계치·기한을 표로 못 박아, 우리가 정할 수 있는 것과 고객만 아는 것을 구분했습니다. 고복잡 정산 도메인의 모호성을 개발 착수 전에 제거합니다.

WHY · 왜 이렇게 설계했나“나중에 정하죠”가 쌓이면 개발 착수 시점에 값이 없어 개발자가 임의로 채웁니다. 파라미터를 표로 성격까지 붙여 두면 — 설계 확정(화면·동작 규칙: PM이 자신 있게 정함)은 즉시 닫고, 현업 확인(실제 마감일·업계 기준값: 고객만 아는 사실)은 기본값을 임시 가정으로 두어 “이 값으로 만들었는데 맞나요?”라고 역제안합니다. 결정이 안 나도 개발이 멈추지 않습니다.
정산 도메인 파라미터 표재현 샘플
항목기본값 (제안)성격근거 · 처리
자료 제출 마감일익월 15일현업 확인임시 가정 — 실제 업무 마감 확인 후 확정, 미결 D-07 연결
미제출 → 재요청 전환마감 +7일설계 확정상태 3단계(수집중→지연→재요청) 화면 동작 규칙
이상값 판정 임계전월 대비 ±20%현업 확인업계 변동 폭은 고객만 앎 — 역제안 후 조정
확정 후 수정 권한관리자만설계 확정확정 = 게이트. 하위 데이터 참조 무결성 보호

AS-IS → TO-BE 흐름 재설계

핸드오프는 화면이 아니라 상태 모델이었습니다 — 데이터가 어떤 상태(임시→검증→확정)를 거쳐 어디로 흐르는지를 먼저 정의하면, 화면·권한·예외 처리 규칙이 그 위에서 자연스럽게 결정됩니다.
