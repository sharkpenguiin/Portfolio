CASE 01 · B2B ERP

제약 유통 B2B ERP
— 기획 전 체계를 설계하다

거래처 5개 유형 × 18개 메뉴를 요구 정의부터 개발 핸드오프까지 단독 기획했습니다.
회의마다 갱신되는 요구는 PRD·결정로그·파라미터 표로 통제했습니다.

고객사 ERP 정보구조도

실제 프로젝트 최종본입니다.
고객사명만 비식별화했고 구조는 그대로 — 로그인부터 시스템 설정까지, 이 트리의 각 메뉴가 SCR-ID 단위 PRD와 1:1로 연결됩니다.

문제

PROBLEM · 01

거래처마다 다른 규칙

5개 유형별로 단가·여신·반품 정책이 제각각.
화면 하나에 모든 예외를 담을 수 없었습니다.

회의마다 뒤집히는 요구

확정한 사항이 다음 회의에서 번복. 무엇이 왜 바뀌었는지 추적할 수 없었습니다.

기획자는 한 명

18개 메뉴를 혼자 설계하며 정합성까지 지켜야 했습니다.

판단

JUDGMENT · 02

화면을 그리기 전에, 규칙을 담을 그릇을 먼저 설계했습니다.

01
예외는 화면이 아니라 파라미터로

거래처 유형별 규칙을 코드에 박지 않고 파라미터 표로 분리. 새 유형이 생겨도 표에 행만 추가하면 되도록 설계.

02
모든 결정에 근거와 출처를

3개월간 20회 이상의 회의에서 바뀌는 요구를 결정로그로 통제했습니다.
결정·근거·출처·상태 라벨을 남겨, 번복돼도 과거 회의록 재검토 없이 근거를 즉시 소명합니다.

03
화면 단위 PRD 체계 — SCR-ID

화면 1개 = 문서 1섹션, 10섹션 구조. 각 화면에 SCR-ID를 부여해 PRD·기능명세·결정로그가 같은 ID로 연결되도록 설계.

04
정합성은 사람이 아니라 체크리스트로

18개 메뉴 간 필드·상태 정합성을 AI 스킬 기반 검증 체크리스트로 통제해 누락을 구조적으로 방지.

산출물

OUTPUT · 03

결정로그·SCR-ID 문서·파라미터 표 샘플 (실제 산출물 기반 재현본 · 고객사·사내 정보 비식별화).

REAL ARTIFACT · 결정로그 실물 발췌 (비식별화)
메뉴별 결정 로그 — "거래처 · 약국" 섹션 원문

· 약국만(도매 분리), 미분류 거래처 → 백그라운드 사업자 조회로 재분류 〔260408,260506〕

· 도매매출로 자동생성, 코드 3시작 6자리, 상태↔개폐업 분리, 가등록(명칭·사업자·주소) 〔260615,260618〕

· 원내약국 = 도매매출 자동 분류(병원 귀속, 등록 시점엔 모름) 〔260618〕

· 🔵 심평원 사업자번호 조회 가능 시 국세청 대체(요양기관 단독)·폐업일자 〔260618〕

· 연계병원 흡수율 섹션 = 혼선 → 제거(실제 요청은 EDI/도매매출 내역 조회) 〔260618〕

줄 끝 〔날짜〕 = 출처 회의. 🔵 = 현업 확인 필요(미결). 회의록은 시간순 원본으로 두고 결정만 메뉴 키로 누적하는 이중 구조라, 명세 작성 시 과거 회의록 전체를 다시 볼 필요가 없습니다.
