반복 업무를 사람이 아니라
시스템이 기억하게

커스텀 AI 스킬 10종SSOT 슬랙 Q&A 봇STT 커스텀 사전5대 검증 기준3계층 메모리 운영

AI를 “빠르게 대신 써주는 도구”가 아니라 설계 대상으로 다뤘습니다. 반복되는 산출물 작업은 커스텀 스킬로 시스템화하고, AI가 만든 결과물은 근거 기반 검증 기준으로 통제합니다. 아래는 그 시스템을 구성 요소별로 재현한 기록입니다. 모든 예시는 비밀유지를 위해 일반화·익명화한 샘플입니다.

개요소스가 검증된 산출물이 되기까지

녹취·회의 같은 원본 소스가 신뢰할 수 있는 산출물이 되는 전 과정을, 사람이 판단하는 지점을 명시해 하나의 파이프라인으로 운영했습니다.

01 · 스킬커스텀 AI 스킬 10종

매번 반복되던 PM 산출물 작업을 스킬로 만들었습니다. 각 스킬은 입력 → 규칙 → 출력이 고정돼 있어, 같은 소스면 같은 품질의 결과가 나옵니다. (스킬명·프로젝트 고유 규칙은 일반화)

스킬 (기능)입력출력
기능명세서 작성회의록·고객 자료화면 단위 명세 시트
명세서 검수작성된 명세 + 원 소스정합성·누락·서식 리포트
회의 사전 준비이전 회의 결정 시드아젠다·확인 질문 목록
회의 사후 정리녹취·메모확정·미결·변경·액션 분류
내부 미팅 정리외부 결과 + 내부 논의공유·결정 통합 페이지
리스크·미결 추출기획서·회의록놓친 항목·선결 결정 목록
고객 전달문 작성진행 상황·요청 사항전달 메일·요약·확인 요청
QA 체크리스트 생성기능 기획·유저 스토리테스트 케이스 목록
인터뷰 질문 가이드새 도메인 개요현황 파악 질문 세트
킥오프 준비 패키지새 프로젝트 정보초기 아젠다·질문·체크리스트

대표 스킬 동작 구조 — 기능명세서 작성

INPUT
소스 문서

회의록·고객 전달 자료·미팅 텍스트. 소스 없이 임의로 내용을 만들지 않는 것이 첫 규칙입니다.

RULES
고정 규칙

소스에 없는 내용 생성 금지 · 등록/수정 필드 대칭 · 필드 유형 일관성 · 출처 태그 부착.

OUTPUT
명세 시트

화면 단위로 요구사항 → 정책 → 필드가 이어지는 구조화된 명세. 이후 검수 스킬로 다시 검증.

라이프사이클원본 관리→패키징(.skill)→배포→버전 관리

02 · 하이라이트SSOT 슬랙 Q&A 봇

팀의 도메인 지식을 하나의 신뢰 소스(SSOT)로 구조화하고, 슬랙에서 질문에 답하는 봇으로 연결한 미니 프로젝트. “봇을 만들었다”가 아니라 “지식 구조와 신뢰도 기준을 설계했다”가 핵심입니다.

MINI CASE · PM이 만든 작은 프로덕트

지식을 사람(PM)이 아니라 시스템이 답하게

팀원의 도메인 질문이 전부 PM을 거쳐야 했습니다. PM이 병목이 되는 구조를, 지식을 SSOT로 구조화해 봇이 대신 답하도록 재설계했습니다.

① 문제
PM = 병목

도메인 질문이 모두 PM 경유. 답이 사람의 기억과 가용 시간에 묶여 있었습니다.

② 설계
SSOT + 신뢰도 기준

지식을 단일 소스로 구조화 → 봇 연결 → 답변의 신뢰도를 통제하는 기준을 정의.

③ 운영 루프
질문 → 개선

실제 질문을 수집해 지식 소스와 답변 품질을 계속 보강하는 루프로 운영.

#도메인-qa · 봇 응답 (가상 도메인)재현 샘플
거래처 정산 자료 마감일이 언제인가요?
익월 15일 기준입니다.출처: 정산 정책 회의(2026-05-26) · 파라미터 표 D-07 — 단, 업계 변동 폭 기준값은 현업 확인 대기(미결)입니다.
단종된 품목도 검색에 노출되나요?
기본은 제외됩니다. 검색 화면의 ‘단종 포함’ 토글로만 노출됩니다.출처: 품목 관리 결정로그(0609 · 내부 확정)
WHY · 신뢰도는 모델이 아니라 큐레이션에서 나온다봇이 잘 답하는 이유는 똑똑한 모델이 아니라, 어떤 지식을 넣느냐에 있습니다. 지식 소스는 Google Drive 문서·Notion·고객사 슬랙 Q&A 채널에서 가져오되, 회의록을 다시 정제하고 녹취·전달 자료·실제 질문답변 같은 1차 소스만 지식으로 씁니다. 틀린 정보가 봇에 들어갈 경로 자체를 소스 단계에서 차단한 것입니다. 그리고 답을 모르거나 미결인 항목은 “확인 대기”라고 정직하게 표시하도록 설계했습니다.

정확도를 숫자 대신 검증 루프로 담보

별도 정답률 지표를 만들지 않았습니다. 대신 봇의 답이 실제로 맞는지를 업무 흐름 안에서 반복 검증했습니다.

STEP 1
실무에 사용

봇 답변을 실제 산출물 작성에 사용.

STEP 2
고객사 미팅에서 검증

그 산출물을 고객사 미팅에서 직접 확인.

STEP 3
반응·검수로 품질 확인

고객 반응과 직접 검수로 틀린 답을 잡아 소스에 반영.

03 · STT커스텀 사전으로 오인식 차단

회의 녹취를 자동 전사할 때 도메인 용어가 자주 오인식됐습니다. 용어 사전을 구축해 전사 단계에서 바로잡고, 근거 추적의 정확도를 높였습니다. (용어는 가상 예시로 익명화)

BEFORE · 사전 없음
“전산 마감일 기준으로…”
“스크린 아이디로 매핑”
“단과 품목은 제외”
AFTER · 사전 적용
“정산 마감일 기준으로…”
“SCR 아이디로 매핑”
“단종 품목은 제외”
오인식 한 단어가 전사 전체의 근거를 무너뜨립니다 — 전사가 틀리면 그 위에 쌓인 회의록·결정·명세가 모두 잘못된 소스를 참조하게 됩니다. 사전은 파이프라인의 가장 앞단에서 이 오류를 막는 장치입니다.

04 · 검증AI 산출물 5대 검증 기준

AI가 만든 결과물을 그대로 쓰지 않기 위한 통제 장치입니다. 모든 산출물은 아래 5개 기준을 통과해야 합니다.

01
근거 검증

모든 항목에 소스가 있는가 — 근거 없으면 AI가 지어낸 것으로 보고 제거.

02
중복 제거

같은 로직·기능이 다른 이름으로 중복 생성되지 않았는가 — 통합하거나 삭제.

03
자동/수동 경계

시스템이 자동 처리할 것과 사람이 판단할 것이 구분돼 있는가.

04
실제 업무 동선

실제 사용자가 일하는 순서·방식과 맞는가 — 화면이 아니라 업무가 기준.

05
데이터 정합

필드가 어디서 와서 어디로 연결되는가 — 출처와 흐름의 정합 확인.

CASE · AI가 임의로 만든 기능을 근거 대조로 제거명세 초안에 원 소스엔 없던 ‘자동 승인’ 옵션이 생성된 적이 있습니다. 회의록·요구 목록을 역방향으로 대조한 결과 근거가 없음을 확인하고 제거했습니다. 그럴듯해 보이는 AI 산출물일수록 “정말 소스에 있는가”를 되묻는 절차가 필요합니다.
다단계 검수변경분 정방향 확인→기존 문서 역방향 모순 스캔→무작위 샘플 소스 대조
이 기준들은 이론이 아니라 실수의 기록입니다 — 한번은 키워드 검색만 믿고 소스를 정리하다, 데이터가 임시 → 검증 → 확정 상태를 거쳐 흐르는 구조를 통째로 놓친 적이 있습니다. 그 뒤로 “기능이 있느냐”만이 아니라 “데이터가 어떤 상태를 거쳐 어디로 흐르는가”를 점검 관점에 추가했습니다. 실수를 개인의 반성으로 끝내지 않고 팀의 규칙으로 바꾸는 것 — 검증 기준이 계속 자라는 방식입니다.

05 · 운영AI 협업을 시스템으로 지속

한 번의 대화로 끝나는 AI가 아니라, 세션이 바뀌어도 맥락이 이어지는 협업 체계를 운영합니다. 3계층 메모리 위에서 AI가 매번 같은 기준으로 일하게 만든 구조입니다.

GLOBAL
전역 원칙

응답 원칙, 커넥터·스킬 환경, 업무 규칙 — 모든 세션에 공통 적용되는 층.

PROJECT
프로젝트 레퍼런스

프로젝트별 결정·용어·구조. 새 세션이 시작돼도 이 층을 읽으면 맥락이 복원됩니다.

SESSION
세션 로그

당일 작업 기록. 자동으로 상위 층에 반영·정리되어 지식이 휘발되지 않게.

일일 자동 보충스케줄로 세션 기록을 매일 메모리에 정리·반영.
작업 규칙 코드화“어떻게 일할지”를 문서로 고정해 세션마다 재현.
AI와의 협업을 세션 단위가 아니라 시스템으로 지속시키는 설계 — 대부분은 대화가 끝나면 맥락이 사라집니다. 메모리 계층을 두면 AI가 어제의 결정을 기억한 채로 오늘 일을 이어갑니다.
